CHAPTER 5

검증 앱 만들기 (정적 · Vercel)

AI가 뽑은 인과관계 해석이 타당한지 사람이 검수하는 앱을 — 서버 없이 정적으로 만들어 Vercel에 올립니다.

이 장의 결과물 https://<프로젝트>.vercel.app — 리뷰어가 샘플별로 ‘적절/부적절’ + 의견을 남기고 적절율(%)로 모델 신뢰성을 수치화하는 정적 웹앱.
왜 정적(static)으로? Streamlit 같은 서버형도 가능하지만(아래 ‘다른 길’ 참고), 우리는 모든 앱을 Vercel로 통일합니다. 검증 앱을 순수 HTML+JS로 만들면 DB·서버 없이 Vercel·Pages 어디든 바로 올라가고, 결과는 브라우저에 저장됩니다.

0. 작업 흐름 — 새 정적 앱을 내 repo로 → Vercel

이번엔 남의 repo를 클론하지 않고 새로 만듭니다(폴더 mab-causal-review/).

PowerShell
# index.html + samples.json 만든 폴더에서
cd mab-causal-review
git init ; git add . ; git commit -m "mAb 정적 검증 앱"
gh repo create mab-causal-review --public --source=. --push
# → 다음 5단계에서 Vercel에 연결

1. 왜 검증인가 — 휴먼 인 더 루프

GPT가 뽑은 관계엔 오류가 섞일 수 있습니다. 모델을 믿기 전에 표본을 사람이 확인해 품질을 수치로 증명합니다. 이 “적절율(%)”이 8장 논문의 신뢰성 근거가 됩니다.

2. 앱 구조 — 단 두 파일

mab-causal-review/
mab-causal-review/
├── index.html      # 검증 UI (CSS·JS 내장, 단일 파일로 동작)
└── samples.json    # 검증 샘플 (인과관계 + GPT 해석)

샘플 하나는 이런 모양입니다 — 인과 엣지 + 근거 + GPT 해석.

samples.json (한 건)
{
  "id": 1,
  "cause": "thermal stress", "relationship": "promotes", "effect": "aggregation",
  "subcat": "physical", "polarity": "neg",
  "evidence": "... exposed hydrophobic patches drive association.",
  "gpt": "열 스트레스가 소수성 패치를 노출시켜 응집을 촉진한다."
}

3. 핵심 동작 — 보고, 판정하고, 저장

서버가 없으니 결과는 브라우저 localStorage에 저장하고, 끝나면 JSON으로 내보냅니다.

index.html · script (발췌)
// 샘플 로드 (samples.json), 없으면 내장 기본 샘플
fetch("samples.json").then(r => r.ok ? r.json() : null)
  .then(data => { if (Array.isArray(data)) SAMPLES = data; })
  .finally(render);

// 판정 저장 → localStorage
function saveCurrent(s){
  reviews[s.id] = { reviewer:name, verdict, feedback, at:new Date().toISOString() };
  localStorage.setItem("mab_review_results_v1", JSON.stringify(reviews));
  next();
}

// 적절율 집계
const good = Object.values(reviews).filter(r => r.verdict==="good").length;
const rate = Math.round(good / Object.keys(reviews).length * 100);
전체 코드 완성된 index.html·samples.json은 mab-causal-review 저장소에 있습니다. 필터·진행률·내보내기까지 포함된 한 파일입니다.

4. 내 데이터로 바꾸기

3장에서 만든 step3_edges.csv의 상위 엣지에 GPT 해석 한 줄을 붙여 samples.json을 채우면 됩니다. 앱의 “JSON 불러오기” 버튼으로 즉시 교체해 미리볼 수도 있습니다.

5. Vercel로 배포

1

로그인

vercel.com에 내 GitHub로 로그인.

2

Import

Add New ▸ Project ▸ Import에서 내 mab-causal-review 선택.

3

설정

Framework = Other(정적), Root = 루트.

4

Deploy

.vercel.app URL 발급. 이후 git push마다 자동 재배포.

6. 결과 집계 — 적절율

앱에서 “리뷰 결과 내보내기”를 누르면 mab_review_results.json이 받아집니다. 적절율은 앱 하단에 실시간으로도 표시됩니다.

이 숫자가 논문이 된다 “리뷰어 검증 결과 적절율 86%” 같은 한 줄이 8장 Results의 신뢰성 절을 채웁니다. 부적절로 표시된 사례의 의견은 3장 프롬프트 개선으로 되먹입니다.

다른 길 — 서버형(Streamlit)이 필요할 때

여러 리뷰어가 중앙 DB에 동시에 기록해야 한다면, 정적 대신 서버형이 낫습니다. 그 경우 STARG-LEE의 mab-review-app(Streamlit + DB)을 내 repo로 가져와 Streamlit Cloud에 배포하면 됩니다(키·DB는 st.secrets). 이번 장의 정적 버전은 localStorage라 한 사람 브라우저에 저장된다는 점만 다릅니다.

혼동 주의 이 정적 앱(mab-causal-review)은 아마존 리뷰용 review-validation과 다른 앱입니다.

오늘의 마무리

  • DB·서버 없이 정적 검증 앱을 만들어 Vercel로 통일 배포했습니다.
  • 판정은 localStorage에 저장하고 JSON으로 내보내기 했습니다.
  • 적절율로 모델 신뢰성을 수치화했습니다.

데이터·시각화·검증이 모두 갖춰졌습니다. 6장에서 이 그래프로 예측 모델 mAb-GATED를 학습시킵니다.
